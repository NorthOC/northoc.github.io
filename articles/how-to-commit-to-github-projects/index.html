Should you commit to Big Github Projects? Yeah, no...

Do you even pull request, bro?

Alright, so at some point, you decide to put on some big boy pants and start making pull requests and commiting to open-source projects to make the world a better place. But I guess this is where the first of many god complexes come to die for many young developers.

Projects are part of the problem

Each open-source project (that matters) has its own pull request standard which can usually be found in some commits.md file or a specific directory, that can be from a couple of pages to dozens of pages in length.

The upsides of this is that you limit spaghetti code and the number of noob devs on the contributor list because such projects assume that you already know how to do all that fancy git magic..

The downsides of this is that you get a small number of contributors, who add actual features. I'm not talking about the dev who do 100s of commits just to reformat the README. No, I'm talking about devs who actually expand on the project further and fix bugs. This is a common bottleneck that bigger projects face and the outcome of this is obvious: when other other projects come to light, these amalgamations will go as they arrived - swiftly. Why? Because time is precious.

A Common Example

For example take the React repo. I'm sure that some of you front-endies are gonna be reeeeing at this, but take a moment to digest this simple fact:

The React repo has over 1500 contributors, and top 6 devs make up for about 50% of all commits. These top contributors are not your standard open-source contributors. They work at Facebook, so it is kind of their job to do this.

The rest of the commits are divided between ~1494 people, which mostly did typo fixes. But before they could submit a fix that took them 2 minutes, they had to go through The Code of Conduct and then The Contributing Guide, spending about 30 minutes of their life, just so that they could feel good.

What's the solution?

To fork the project, change the contribution standards and claim it your own.

Just kidding. The solution is to be mindful of these practices, notice them early and avoiding projects that employ complex commit schemes like the plague.

The simple way to commit

Heres a simple way to do this stuff (the green lines are cli commands):

  1. Fork the repository you want to contribute to

  2. Clone the fork to your pc with git clone https://github.com/your_username/your_repository name

  3. Create a branch for your feature with git checkout -b branch_name

  4. Add the branch upstream so that github has access to it git remote add upstream https://github.com/your_username/your_repository_name

  5. Edit the code

  6. git add .

  7. git commit -m "message of what you've changed"

  8. git push

Once this is done, open your fork in the browser. There should be two buttons above the files saying "Contribute" and "Sync Fork". Click the contribute button and then "Open pull request". Write what you've changed, submit it and let the project jannies do the rest of the work.

That's the gist of it, but git is way more complex than this. So take your time to learn a bit more about resolving conflicts and how the HEAD works.

Congrats! You now have the ability to make the world a slightly better place.
